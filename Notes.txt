---------------- Config File Functionalities ----------------
### Global Server Setting:
	- server { ... } block: This defines a "Virtual Server." In HTTP/1.1, a single physical machine can host multiple websites. 
	Your server will read the Host header in the HTTP request to determine which server block should process the request.

	- listen: Specifies the port the server will listen on for incoming TCP connections (e.g., 8080).

	- host: Specifies the IP address to bind to. 127.0.0.1 means it only accepts connections from the local machine. 
	0.0.0.0 means it accepts connections from any network interface. 
	(Note: Standard NGINX often combines this into listen 127.0.0.1:8080;, but keeping them separate in webserv makes parsing easier).

	- server_name: Defines the domain names associated with this server block. If a request comes in on port 8080
	with the header Host: example.com, the server matches it to the block containing server_name example.com.

	- client_max_body_size: Crucial for security and subject requirements. It sets the maximum allowed size of the client
	request body (usually for POST requests). If the Content-Length header in a request exceeds this size,
	your server should immediately respond with a 413 Payload Too Large error.

	- error_page: Defines a custom HTML file to serve when a specific HTTP error occurs. For instance, error_page 404 /404.html;
	tells your server to read and return the 404.html file instead of a hardcoded string when a resource is not found.


### Directory & Routing Settings:
	- root: Defines the base directory for requests. If root /var/www/html; is set, a request to http://localhost:8080/style.css 
	will cause your server to look for the file at /var/www/html/style.css on your hard drive.

	- index: Specifies the default file to serve if the client requests a directory. If the request is for /, the server 
	appends the index file, turning the internal request into /index.html.

	- location <path> { ... } block: Defines how to process requests matching a specific URI pattern. Your server must find 
	the most specific location match for the incoming URI.

What does inet_addr() and htons()?
- inet_addr(): We use this function to convert char * C strings into in_addr_t, that's why we casted the
               std::string into .c() first. We use the in_addr_t format for sending packets with TCP/IP
			   through internet, because the router doesn't have time to check the size of each packet,
			   we use this format as an standard, a 32-bit unsigned integer. This function also transform
			   the bits into big endian (the larger bits for binaries are at the beginning, while Macs and
			   Linux often use little endian, where the smaller bits in binary notation are at the beginning)
			   so it makes sure that the bits are arranged for universal understanding across machines.

- htons(): This function does something similar to inet_addr(), it swaps the bytes into Network Byte order
           to make an standard across machines and make possible the communication. That's why in the project
		   we transform the std::string for the port into .c() and then make std::atoi(), we are already
		   passing to the function an unsigned 16-bits int as argument.

















Sockets in C:
- why using?: socket(AF_INET, SOCK_STREAM, 0);
	- AF_INET: It just tells the "socket family" to use, which means which protocol use for the socket, which
	           also means chosing between IPv4 or IPv6. With AF_INET we are telling to use IPv4, with AF_INET6
			   we are telling to use IPv6 socket family.

	- SOCK_STREAM: With this we are telling the OS that we want a TCP connection. Which means that we want a
	               bidirectional stream of communication (or data interchange). Another alternative is using
				   SOCK_DGRAM for UDP protocol, but this protocol doesn't validates that all the packets are
				   recieve and it's not reliable for a webpage but it is for a multiplayer game.
	
	- 0: This tells the OS tho chose the default protocol, so as we said to use AF_INET and SOCK_STREAM, the OS
	     would know that we are talking about TCP Protocol under the hood.

- how to make the sockets non-blocking as the subject says? -> We use fcntl(sockets[i].getSocketFD(), F_SETFL, O_NONBLOCK):
	- fcntl(): As everything in Linux is a file, we use this function from C in the User Space to make the Kernel

- how to make possible for the user to reach the server using the browser -> bind(sockets[*i].getSocketFD(), (struct sockaddr *)&addr, sizeof(addr)):

Main loop event of the server -> initServer():
A real server doesn't goes one by one of the client's FD checking if there is something new requested by them, that
would be extremely costly for the CPU, spyking it's performance to 100%. Instead of that we use another aproach, we
wait until getting a new update from one of the FDs. That's what poll() is in charge for.

In reality, poll() is a multiplexer, specifically, the multiplexer that we are going to use because is compatible with Linux and Mac.

Multiplexers are a System Call Routine (different from Kernel Objects like sockets or pipes), which are a set of instructions inside
the CPU in priviledge ring 0, which means that they are managed by the Kernel.

We create a pollfd for each socket, there are X servers and each server has a socket, so we create at first X ammount of pollfd

Then we execute poll(&this->_pollFDs[0], this->_pollFDs.size(), -1):
	- &this->_pollFDs[0]: The memory address where for the array of pollfd, it points to the beginning.

	- this->_pollFDs.size(): The size of the array of pollfd

	- -1: Set the time to wait in miliseconds, as is set to -1 it is always waiting until POLLIN
